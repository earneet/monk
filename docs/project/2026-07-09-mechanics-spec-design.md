## 3. 总则

### 3.1 记号
- `P` = 玩家路径,有序格子序列 `[c₀, c₁, …, cₖ]`
- **`len(P)` := P 中格子数**(序列长度 = k+1;即「步数 + 1」)。GDD §7.7「已走步数 `len(P)`」以 `len(P)`(格子数)为基准
- `lever_cell(id)` = 机关 `LeverData.id` → 其坐标的映射(LevelSystem 加载时建立)

### 3.2 查询接口
- **`can_pass(coord, P) -> bool`**:机制是否允许踏入 `coord`。**仅查机制通行性**,不含「不重复」校验
- **`state(coord, P)`**:该格的派生状态(门开闭 / 桥铺放 / 水位)

### 3.3 PathState.move 的校验(三层入口校验 + 传送追加)
`PathState.move(coord)` 接受移动意图时,依次做**入口校验**:
1. **不重复**:`coord ∉ P`(GDD 核心规则)
2. **邻接**:`coord` 必须空间邻接 `path[-1]`(网格四邻接一步;起点首步除外)
3. **机制通行**:`can_pass(coord, P)` 为 true

入口校验通过则 `coord` 入 path。**传送强制**不在入口校验,而由踏入口后的**自动追加**实现(见 §6):若 `coord` 自身是传送门入口 X,则 move 紧接着追加其配对端 Y(强制传送步,玩家无需再输入)。即传送的强制性 = 自动追加(踏入入口即被强制完成传送),move **不做**「path[-1] 是传送门则 coord 必须配对端」的入口判断——该判断与自动追加冲突(自动追加后 path[-1]=出口 Y 也是传送门,会误伤合法的离开步)。

> 修订(2026-07-10,传送批次 2):本节早前把传送列为「第三层入口校验(若 path[-1] 是传送门则强制配对端)」,与 §6 自动追加存在张力。澄清后传送归自动追加(§6),入口三层 = 不重复 / 邻接 / 机制通行,与 `scripts/grid/path_state.gd` 实现一致。

### 3.4 状态确定性
所有派生状态(门开闭、机关被踩、桥铺放、动态水水位)均为 `P` 的纯函数(见架构 §9);撤销 = 截短 `P`,状态自动回滚,零副作用。

## 4. 机制规范

### 4.1 假山(WALL)
- **数据**:`TileType.WALL`(枚举),无字段
- **`can_pass`**:恒 `false`
- **状态**:恒定(永久障碍)
- **计入需扫**:否
- **边角**:—

### 4.2 流水·静态(FLOWING_WATER)
- **数据**:`TileType.FLOWING_WATER`(枚举),无字段
- **`can_pass`**:恒 `false`
- **状态**:恒定(永久障碍;与动态水 §4.7 区分——后者在 `mechanics` 列表)
- **计入需扫**:否
- **边角**:—

### 4.3 门(DoorData)
- **数据**:`coord`, `lever_ids: Array[String]`
- **`can_pass`**:门开
- **状态(门开)**:`∃ id ∈ lever_ids, lever_cell(id) ∈ P`(OR 语义,任一机关被踩即开)
- **计入需扫**:是
- **边角**:`lever_ids` 为空 → 门永关;数据校验期拒绝空 `lever_ids`(见 §7)

### 4.4 机关(LeverData)
- **数据**:`coord`, `id: String`
- **`can_pass`**:恒 `true`(机关格可通行)
- **状态(被踩)**:`coord ∈ P`
- **计入需扫**:是
- **边角**:—(机关无独立状态,被踩由 P 决定)

### 4.5 传送门(PortalData)
- **数据**:`coord`, `pair_id: String`
- **`can_pass`**:`true`(可踏入;不重复由 PathState)
- **强制传送(双向,归 PathState.move)**:踏入任一端 X 后,下一步**必须**是其配对端 Y(同 `pair_id`)。X、Y 各踩一次(哈密顿每格一次天然保证不重复)
- **状态**:配对关系恒定(由关卡配置)
- **计入需扫**:是(X、Y 各一次)
- **边角**:`pair_id` 无配对(孤立)或配对数 ≠ 2 → 数据校验期拒绝(见 §7)

### 4.6 桥(BridgeData)
- **数据**:`coord`, `lever_ids: Array[String]`
- **`can_pass`**:桥铺放
- **状态(铺放)**:`∃ id ∈ lever_ids, lever_cell(id) ∈ P`(OR 语义)
- **计入需扫**:是
- **边角**:同门(`lever_ids` 为空 → 永未铺放;校验期拒绝,见 §7)

### 4.7 动态水(DynamicWaterData)
- **数据**:`coord`, `period: int`(默认 4)
- **相位**:`phase = len(P) mod period`(P 为踏入 coord 前的当前路径)
- **水位**:`LOW ⟺ phase < (period + 1) / 2`(整数除法,即 `phase < ceil(period/2)`);`LOW` = 落(可通行),`HIGH` = 涨(不可通行)
- **`can_pass`**:`level == LOW`
- **计入需扫**:是(须在 LOW 相位扫过)
- **边角**:`period` 必须 ≥ 2(校验期拒绝 `< 2`,见 §7)

**相位分布表(落 ≥ 涨)**:

| period | LOW 相位(落,可通行) | HIGH 相位(涨,不可通行) |
|---|---|---|
| 2 | 0 | 1 |
| 3 | 0, 1 | 2 |
| 4 | 0, 1 | 2, 3 |
| 5 | 0, 1, 2 | 3, 4 |

> LOW 占 `ceil(period/2)` 相,始终 ≥ HIGH,契合休闲直觉(动态水大部分时间可通行,涨水为短暂阻碍)。此相位划分为本规范补充(GDD §7.7 仅规定「涨 / 落」,未定占比)。

## 5. 需扫格汇总(遵循数据格式 §9)

非永久障碍格全部计入需扫:`EMPTY` + 门 + 机关 + 传送 + 桥 + 动态水;假山 / 流水不计。

## 6. 传送的路径构造(PathState.move 完整逻辑)

玩家移动到传送门时的完整流程:
1. 玩家意图移动到 `coord`(传送门 X)→ move 校验三层(§3.3)通过 → `X` 入 path
2. move 检测 `X` 是传送门 → 自动追加其配对端 `Y` 到 path(强制传送步;`Y` 不需玩家再次输入)
3. 此后玩家从 `Y` 继续(`Y` 的下一步走 `Y` 的空间相邻格)

> 即传送 = 两步入 path:`X`(玩家主动踏入)+ `Y`(系统强制追加)。两者均计入需扫。

## 7. 数据校验(由关卡设计工具保证)

关卡 `.tres` 加载时(或设计期)须满足以下校验,否则视为无效关卡(拒绝加载 / 设计期报错):

- **`DynamicWaterData.period ≥ 2`**(`= 0` 除零、`= 1` 退化为静态且语义歧义)
- **`DoorData` / `BridgeData.lever_ids` 非空**(空 → 门永关 / 桥永不放,格计入需扫则关卡必无解)
- **`PortalData.pair_id` 恰好成对**(每个 `pair_id` 在全关恰好出现 2 次;孤立或 > 2 拒绝)
- **`lever_ids` 引用的机关 id 存在**(引用不存在的机关 → 无效)
- **关卡可解**:存在至少一条满足胜利条件(覆盖全部需扫格 + 可选终点)的哈密顿路径

> 上述校验由**关卡设计工具**(用户规划,见 §9)在设计期保证,并提供可解性验证。运行时(LevelSystem.load)做轻量校验(字段有效性),可解性由设计期保证(GDD §5:不提示死局)。
