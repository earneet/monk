## 1. 背景与目标

## 3. 游戏概述

操控小和尚,在寺院网格上走出一条**扫过所有可通行土地、不重复**的路径(网格上的哈密顿路径问题)。

- 类型:休闲益智 / 网格一笔画
- 平台:桌面 + 移动(双输入)〔②〕
- 核心体验:自由探索式解谜,**可撤销、无失败**,鼓励试错〔①〕

## 4. 核心规则

- **网格**:规则矩形,**四向正交移动**(上下左右,不斜向);玩家占据一格,每步移动到正交相邻格
- **不重复**:不可踏上已扫过的格(撤销除外)
- **可撤销**:逐步回退已走路径;撤销永不构成失败,且**彻底消除该步的一切影响**(见下方确定性原则)
- **状态确定性原则(地基)**:所有可变机制状态——门开闭、机关触发、桥铺放、动态水水位——均为**玩家路径的纯函数**,系统不存在独立于路径的状态机。
  - 后果:撤销 = 缩短路径,所有派生状态自动回滚,零副作用、可重放、可验证。
  - 这条原则贯穿机制系统设计,使「可撤销」在含复杂机制的关卡中依然自洽。

### 路径模型(状态确定性的形式化预览)

设玩家路径为有序格子序列 `P = [c₀, c₁, …, cₖ]`(c₀ 为起点)。任意时刻的全部游戏状态由 `P` 唯一确定:

- 已扫格集合 = `{c₀, …, cₖ}`
- 门 `d` 开启 ⟺ `P` 包含 `d` 的**任一**控制机关格(OR 语义,任一机关被踩即开)
- 桥 `b` 铺放 ⟺ `P` 包含 `b` 的**任一**控制机关格
- 动态水水位 = `f(len(P) mod 周期)`
- 传送:若 `cᵢ` 是传送门 A,则 `cᵢ₊₁` 须为其配对 B(传送步)

撤销即将 `P` 截短为 `[c₀, …, cₖ₋₁]`,上述全部状态自动重算。完整形式化留「机制规范」文档。

## 5. 胜负

- **胜利**:当前路径 `P` 覆盖关卡全部「需扫格」〔③〕
- **需扫格** = 关卡中所有「非永久障碍」的格:
  - 永久障碍(不计入需扫):假山、静态流水
  - 计入需扫:普通可通行格、门格、机关格、桥位、传送门、动态水底格等一切「有条件或无条件可通行」的格
- **可选终点**:关卡数据可定义指定终点格;若定义,胜利还须 `P` 末端 == 终点格〔③〕
- **失败**:**无**(可撤销设计下去除失败概念)。游戏状态仅有「未完成 / 已完成」
- **死局**:玩家走入「无路可走且未完成」时,**完全不提示**;玩家自行察觉并撤销重试

> 关卡可解性由关卡设计者保证(设计时确保存在至少一条满足条件的哈密顿路径)。可解性自动校验工具列为未来扩展(见 §12)。

## 6. 移动与操作〔②〕

- **桌面**:方向键 / WASD,逐格移动
- **移动端**:点击相邻可通行格,移动一步
- **撤销**:独立「撤销」按钮(逐步回退);亦允许点击来路格回退
- **重置**:一键清空路径回到起点(冗余但便利,因撤销已能逐步回到起点)

## 7. 机制(原型期全部实现)〔⑤〕

每条机制给出:**行为 / 是否计入需扫 / 状态如何随路径变化**(均符合状态确定性原则)。

### 7.1 假山
- 永久障碍,不可通行、免扫
- 计入需扫:**否**
- 状态:恒定(永久)

### 7.2 流水(静态)
- 永久障碍,不可通行、免扫
- 计入需扫:**否**
- 状态:恒定(永久);与动态水(§7.7)区分

### 7.3 门
- 默认关闭、不可通行;由机关开启;开启后可通行可扫
- 计入需扫:**是**(开启后须扫过)
- 状态:**门 d 开启 ⟺ 路径 P 已包含 d 的任一控制机关格**(触发式:踩过即永久开,撤销到踩机关之前则关回)

### 7.4 机关
- 玩家踩到即触发,开启其关联的门 / 桥(可 1 对多)
- 计入需扫:**是**(玩家须踩过)
- 状态:由路径包含关系决定(见 §7.3 / §7.6)

### 7.5 传送门
- 成对出现(A, B);**强制传送**——踩入任一端 → 下一步出现在另一端(传送步:空间不相邻,但逻辑相邻,仍算路径的一步,不可选择走空间相邻格)
- 计入需扫:**是**(A、B 均须踩过,各一次)
- 状态:传送配对关系恒定(由关卡配置);是否在路径上由 P 决定。因哈密顿路径每格一次,A、B 各踩一次,天然不构成循环

### 7.6 桥
- 横跨水域的可通行构造,默认未铺放、不可通行;由机关放下;放下后该格可通行可扫
- 计入需扫:**是**(铺放后须扫过)
- 状态:**桥 b 铺放 ⟺ 路径 P 已包含 b 的任一控制机关格**

### 7.7 动态水
- 周期性涨落;**水位 = f(已走步数 len(P) mod 周期)**
- 涨 = 不可通行;落 = 可通行可扫
- 计入需扫:**是**(须在低水位时扫过)
- 状态:由路径长度确定性推导;玩家可通过控制步数选择水位(额外的解谜维度)

## 8. 关卡与进度

### 8.1 关卡数据
- 格式:`.tres`(Godot Resource),可编辑器可视化编辑、利于版本管理与批量制作
- 内容:网格布局、各格机制类型与配置、起点、(可选)终点、机关 ↔ 门 / 桥映射、动态水周期等
- 详细字段设计留独立「关卡数据格式」文档

### 8.2 进度结构〔④〕
- **线性主线**:必通关卡顺序解锁
- **可选分支 / 隐藏关**:穿插于主线,提供探索感
- **章节化组织**(基于大内容量愿景 §9 的推论):主线按「章节 / 世界」分组(如每 ~10 关一章,主题如前院 / 后山 / 藏经阁);章节间线性,章节内「线性主线 + 可选分支」
  - 理由:40~50+ 关完全扁平线性会过长,章节分组改善导航与主题节奏
  - **待确认**:若你更想要完全扁平线性(无章节),可调整

### 8.3 进度持久化
- 玩家通关进度须持久化(已完成关卡、解锁状态)
- 原型期:本地持久化(如 `user://` 下的存档文件);云端同步列为未来扩展

## 9. 内容量与范围

### 9.1 内容量愿景〔用户决策〕
- 目标:**大规模、可持续扩展**的关卡体系(起步 40~50 关,设计上支持无限增长,运营向,类似休闲消除类)
- 支撑:机制数据驱动 + `.tres` 关卡格式 + 章节化组织,使批量制作与持续更新低成本

### 9.2 原型期 scope(YAGNI)
- **纳入**:§7 全部机制 + 章节化线性主线 + 分支 + 可撤销 + 双输入 + 本地进度持久化 + 少量验证关卡(约 5~10 关,用于验证核心循环)
- **暂不纳入**:星级评价、收集物、计时、多人、关卡编辑器、成就系统、云端同步、内购
- **扩展位**:机制数据驱动设计天然支持未来新增机制(传送 / 桥 / 动态水之外的新类型)
