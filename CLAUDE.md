# monk —— 一休扫地

休闲益智类「一笔画」游戏。玩家操控小和尚,需扫过寺院中每一寸可通行土地,
且不可重复踏上已扫过的格子(本质是网格上的哈密顿路径问题)。

核心机制:
- 障碍:流水、假山 —— 不可通行、免打扫
- 门:可穿行;部分门需先踩机关才能开启
- 机制会持续扩展(传送、桥、动态水等),故机制系统设计为数据驱动、可插拔

当前阶段:原型期(仅 Scenes/boot.tscn,尚无业务代码)

## 技术栈

- 引擎:Godot 4.7(stable)
- 语言:纯 GDScript(单一语言,不混用其他脚本语言)
- 渲染管线:Forward+
- 注:addons/py4godot 为早期实验遗留,非本项目所用,待移除(见「已知技术债」)

## 目录结构

类型分层组织:

    monk/
    ├── project.godot
    ├── scenes/            # 所有 .tscn(按子系统命名:boot / grid / level / ui …)
    ├── scripts/           # 所有 .gd,按子系统分子目录
    │   ├── grid/          # 网格模型、瓦片类型、路径校验
    │   ├── mechanics/     # 各类机制,每个机制一个 .gd
    │   ├── level/         # 关卡加载、进度、胜负判定
    │   └── ui/            # HUD、菜单
    ├── resources/         # 关卡数据 .tres
    ├── assets/{art,audio,fonts}
    └── addons/            # 第三方插件(py4godot 待移除)

约定:场景入 scenes/,脚本入 scripts/ 对应子目录;脚本与同名场景配对。

## 架构原则

- 机制数据驱动:每种地形/机关 = 一份数据定义(Resource/脚本)+ 一段规则逻辑,
  新增机制不改主循环。
- 逻辑/表现分离:规则逻辑(能否通行、触发什么)独立于渲染表现(贴图、动画),
  便于测试与替换。
- 关卡数据用 Resource(.tres):关卡布局与机制配置序列化为 .tres,可在编辑器
  可视化编辑、利于版本管理。
- 单一职责:每个脚本只做一件事;文件膨胀即为重构信号。

## 代码规范

- 命名:变量/函数 snake_case;类名/节点名 PascalCase;常量 UPPER_SNAKE;
  私有成员加 _ 前缀。
- 类型:尽量使用静态类型注解(var x: int = 0、func f() -> void:)。
- 注释:严格承接全局 CLAUDE.md —— 默认不加注释,只在必要时解释「为什么」
  (业务规则、非常规操作、复杂算法);每函数不超过 3 行注释;禁止「描述做什么」
  的注释。
- 每个脚本用 class_name 暴露类型,避免全局命名冲突。
